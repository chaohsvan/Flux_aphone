# Flux 功能特性详解

本文档用于系统描述 Flux 当前已经形成的功能特性、交互规则、数据规则和边界情况。它不是技术栈说明，也不是视觉设计说明，而是面向产品验收、开发维护和后续迭代的功能规格手册。

## 1. 功能总览

Flux 是一个本地优先的个人效率与生活记录系统，核心由四类业务能力组成：

| 功能域 | 核心目标 | 主要页面 | 关键数据 |
| :--- | :--- | :--- | :--- |
| 日记 | 记录每天的正文、心情、位置、天气、附件和标签 | 日记页、日历页、设置附件管理 | `diaries`, `diary_tags`, `diary_tag_links`, `diaries_fts`, `data/attachments` |
| 待办 | 管理任务、子任务、优先级、截止时间、完成状态和历史 | 待办页、日历页 | `todos`, `todo_subtasks`, `todo_projects`, `todo_history` |
| 日历 | 按时间聚合事件、日记、待办、节假日和回收站内容 | 日历页 | `calendar_events`, `calendar_holidays`, `calendar_static_holidays` |
| 数据治理 | 处理导出、附件清理、回收站、恢复和数据一致性 | 日记页、待办页、日历页、设置页 | 软删除字段、恢复字段、导出接口、附件扫描 |

当前系统的功能原则：

- 所有核心数据默认保存在本地 SQLite 和本地附件目录中。
- 页面显示状态和业务事实分离，例如“日历隐藏待办图层”不代表删除待办。
- 删除优先采用软删除，便于回收站恢复。
- 日历是跨模块聚合视图，日记、待办和事件仍分别归属自己的业务域。
- 日记遵循“一天一篇主日记”的产品模型。
- 待办完成后仍保留在原列表中，只以已完成状态置底。

## 2. 日记功能

### 2.1 日记主模型

日记是 Flux 的核心记录对象。每篇日记包含以下信息：

| 字段 / 信息 | 说明 |
| :--- | :--- |
| 日期 | 日记所属日期，是日记最重要的时间维度。 |
| 时间 | 可选，手动记录具体时间；为空时只按日期展示。 |
| 正文 | Markdown 正文，是日记主体内容。 |
| 心情 | 可选，用于筛选和回顾。 |
| 天气 | 可选，手动记录当天或写作时的天气。 |
| 位置 | 可选，手动记录地点。 |
| 标签 | 可多个，用于主题归类。 |
| 收藏 | 用于快速筛选重要日记。 |
| 附件 | 通过 Markdown 引用图片、音频或普通文件。 |
| 字数 | 后端根据正文自动计算。 |
| 删除状态 | 通过 `deleted_at` 判断是否进入回收站。 |
| 恢复状态 | 通过 `restored_at` 和 `restored_into_id` 记录合并恢复关系。 |

产品上日记不再强调标题。系统仍会在内部保留标题字段以兼容旧数据和列表展示，但用户写作体验以正文为中心。

### 2.2 一天一篇主日记

Flux 当前规则是同一天最多存在一篇未删除的主日记。

实现规则：

- 新建日记时，如果目标日期没有未删除日记，则创建新记录。
- 新建日记时，如果目标日期已经存在未删除日记，则转为更新当天原日记。
- 编辑日记时，如果把日期改到另一个已有日记的日期，系统会拒绝冲突。
- 数据库使用部分唯一索引兜底，防止绕过前端产生同日多篇未删除日记。

这个设计让用户在日历和日记归档中看到的是“按天归档”的生活记录，而不是同一天分裂成多个主条目。

### 2.3 日记编辑体验

日记编辑页的目标是接近“写一篇文章”的体验，而不是填写一张表单。

当前能力：

- 正文区域更大，优先承载连续写作。
- 日期、时间、心情、天气、位置与标签作为辅助元数据。
- 支持 Markdown 语法。
- 支持插入本地图片、本地音频和普通文件附件。
- 支持收藏标记。
- 保存按钮放在编辑区域上方，减少长文编辑后的滚动成本。

编辑行为：

- 用户点击新建或编辑后进入日记编辑面板。
- 保存时前端提交正文和元数据。
- 后端负责校验日期唯一性、更新标签关联、刷新全文索引。
- 保存成功后前端刷新日记列表、标签筛选和日期归档。

### 2.4 Markdown 与附件引用

日记正文支持 Markdown。附件不是直接嵌入数据库，而是上传到本地附件目录后，以 Markdown 链接形式写入正文。

支持的引用形式：

```markdown
![图片](/attachments/diaries/2026/04/example.png)
[audio:录音](/attachments/diaries/2026/04/example.mp3)
[file:文档](/attachments/diaries/2026/04/example.pdf)
```

规则：

- 图片用于正文内视觉记录。
- 音频用于记录录音、声音片段或口述内容。
- 普通文件用于 PDF、文档、表格、压缩包等补充材料。
- 所有附件必须是本地文件上传，不依赖网络图片。
- 附件大小必须小于 100MB。

### 2.5 日记筛选

日记筛选分为四个大类：

| 筛选类目 | 说明 |
| :--- | :--- |
| 收藏 | 只看收藏日记。 |
| 心情 | 按心情值筛选。 |
| 月份 | 按 `YYYY-MM` 聚合和筛选。 |
| 年份 | 按 `YYYY` 聚合和筛选。 |

交互规则：

- 大类以按钮形式放在一起。
- 下方只展示当前选中大类的筛选项。
- 每个筛选类目下的结果按日期排序。
- 筛选只影响日记页展示，不修改日记数据。

### 2.6 标签与日期视觉强化

日记页面会强化日期与标签的视觉元素，用于提升回顾效率。

目标：

- 日期是日记列表中最重要的主视觉信息之一。
- 标签用于表达主题和归类，不应被淹没在正文摘要里。
- 用户能快速判断一篇日记属于哪一天、哪些主题、是否收藏。

典型展示：

- 日期以更醒目的位置和样式展示。
- 标签以紧凑的小元素展示。
- 收藏、心情、天气、位置等元数据辅助呈现。

### 2.7 关键词搜索与全文检索

日记支持关键词搜索，覆盖正文和元数据。

可检索范围：

- 正文 Markdown
- 日期
- 时间
- 心情
- 天气
- 位置
- 标签

实现逻辑：

- 主要使用 SQLite FTS5 进行全文检索。
- 同时使用 `LIKE` 作为补充，覆盖短文本、日期和中文分词不理想的场景。
- 删除的日记不会进入普通搜索结果。
- 日记创建、更新、删除和恢复时会同步维护全文索引。

前端表现：

- 搜索框输入关键词后返回匹配日记。
- 结果可用于全局关键词定位。
- 编辑器中可以围绕关键词进行定位和高亮。

### 2.8 日记导出

日记支持多选导出。

交互规则：

- 点击“导出 / 多选”后进入选择模式。
- 默认全选当前结果中的日记。
- 用户可以逐条取消选择，也可以全选、反选。
- 选择完成后再选择导出格式。

支持格式：

| 格式 | 用途 |
| :--- | :--- |
| JSON | 保留结构化字段，适合备份和二次处理。 |
| CSV | 适合在表格工具中筛选、统计和整理。 |
| Markdown | 适合阅读、迁移和长期归档。 |

导出内容：

- 日期、时间
- 心情、天气、位置
- 收藏状态
- 标签
- 正文
- 创建和更新时间

注意事项：

- Markdown 导出会保留 `/attachments/...` 引用。
- 附件二进制文件当前不会被自动打包进导出文件。

### 2.9 日记删除与回收站

日记删除采用软删除。

删除时：

- 日记不会立即从数据库物理删除。
- 系统写入 `deleted_at`。
- 普通日记列表不再展示该日记。
- 日记全文索引中移除该日记，避免搜索命中。
- 日历在开启回收站图层时可以显示删除标记。

回收站中：

- 可以查看已删除但尚未恢复合并的日记。
- 可以执行恢复。
- 已经合并恢复到另一篇日记的来源记录不再作为可恢复项反复出现。

### 2.10 日记恢复机制

日记恢复是当前系统中最重要的复杂规则之一。

恢复时分两种情况：

```text
恢复某篇已删除日记
  |
  v
检查该日期是否已有未删除日记
  |
  |-- 没有：直接恢复原日记
  |
  |-- 有：增量合并到该日期现有日记
```

直接恢复规则：

- 清空 `deleted_at`。
- 清空恢复相关标记。
- 刷新全文索引。
- 日记重新出现在列表和日历中。

合并恢复规则：

- 删除日记的正文追加到当天现有日记正文后面。
- 标签合并去重。
- 如果现有日记缺少时间、心情、天气、位置等元数据，则用被恢复日记补齐。
- 收藏状态采用合并逻辑，只要其中一篇收藏，合并后保持收藏。
- 被恢复的来源记录写入 `restored_at` 和 `restored_into_id`。
- 来源记录不再作为可恢复日记重复出现在回收站。
- 目标日记刷新全文索引。

这样处理可以解决“一天有多篇删除日记，恢复时不应破坏一天一篇主日记”的问题。

### 2.11 日记在日历中的显示

日记可以显示在日历页面，但默认不显示。

规则：

- 日历页面有日记图层按钮。
- 默认状态下日记图层为取消状态。
- 用户点击日记图层后，日历开始渲染日记标记。
- 再次点击后隐藏日记标记。
- 图层开关只影响展示，不影响日记数据。

在日期详情中：

- 如果当天有日记，会显示日记卡片。
- 如果同时有事件、待办、日记，页面会用不同类型的视觉标识区分。
- 日记删除后，普通图层不显示；回收站图层开启时可以显示删除标记。

## 3. 待办功能

### 3.1 待办主模型

每个待办任务包含以下主要信息：

| 信息 | 说明 |
| :--- | :--- |
| 标题 | 必填，任务的核心描述。 |
| 描述 | 可选，任务详情。 |
| 状态 | 未完成、进行中或已完成等状态。 |
| 优先级 | 当前收敛为正常和高优先级两档。 |
| 标签 | 用于归类，历史命名中仍有 `project` 字段。 |
| 截止时间 | 用于近期、逾期、定时任务和日历展示。 |
| 开始时间 | 可选，辅助记录任务周期。 |
| 提醒字段 | 可保存提醒分钟数，但当前不实现真实系统通知。 |
| 子任务 | 拆分任务步骤。 |
| 排序值 | 支持拖拽后的自定义排序。 |
| 历史记录 | 记录创建、编辑、完成、恢复、删除等动作。 |
| 删除状态 | 通过 `deleted_at` 进入回收站。 |

### 3.2 优先级规则

优先级只保留两档：

- 正常
- 高优先级

历史上“标记为重要”和“高优先级”已合并为同一个功能语义。

规则：

- 高优先级任务在智能排序中优先靠前。
- 批量操作中可以把选中的任务设置为高优先级。
- 批量操作中也可以把选中的任务设置为正常优先级。
- 数据层仍兼容 `is_important` 字段，但产品语义以优先级为准。

### 3.3 完成与重新打开

每个任务都有完成按钮。

交互规则：

- 未完成任务点击完成按钮后，状态变为已完成。
- 已完成任务再次点击完成按钮后，状态恢复为未完成。
- 不再提供单独的撤销按钮。
- 完成状态切换会写入任务历史。

显示规则：

- 已完成任务不会移动到独立的已完成列表。
- 已完成任务仍保存在原标签、原列表和原筛选上下文中。
- 已完成任务通过排序置底。
- 已完成任务视觉上变为灰色。

### 3.4 子任务

子任务支持独立完成。

规则：

- 子任务有自己的完成按钮。
- 未完成子任务点击后变为完成。
- 已完成子任务再次点击后恢复未完成。
- 子任务完成后保持原位置，不需要置底。
- 子任务完成状态不自动决定父任务是否完成。

### 3.5 待办列表与智能排序

待办支持多种筛选：

- 按标签
- 按状态
- 按优先级
- 按关键词
- 按近期
- 按逾期
- 按定时任务
- 按自定义日期范围

智能排序优先级：

```text
未完成任务
  -> 高优先级
  -> 有截止时间
  -> 截止时间更早
  -> 手动 sort_order
  -> 创建时间更晚或更早的配置排序
已完成任务置底
```

这个排序的目标是让“更需要行动的任务”自然靠前，同时不丢失用户通过拖拽形成的局部顺序。

### 3.6 拖拽排序

待办支持拖拽排序。

规则：

- 用户拖动任务后，前端提交新的顺序。
- 后端保存 `sort_order`。
- 智能排序中 `sort_order` 不是唯一依据，而是在优先级、截止时间等因素之后参与排序。
- 已完成任务仍会被整体置底。

### 3.7 待办标签

待办的“项目”概念已在界面上改为“标签”。

说明：

- UI 使用“标签”。
- 部分历史数据库表和 API 仍保留 `todo_projects` / `project_id` 命名。
- 标签可以用于筛选任务。
- 删除标签时，关联任务会解除标签，而不是删除任务。

### 3.8 待办详情历史

每个任务可以查看详情历史。

记录内容包括：

- 创建任务
- 编辑任务
- 完成任务
- 重新打开任务
- 删除任务
- 恢复任务
- 批量设置优先级

用途：

- 帮助用户理解任务状态变化。
- 帮助开发和测试追踪任务行为。
- 为后续同步或审计功能预留基础。

### 3.9 待办导出与多选

待办支持多选模式。

交互规则：

- 点击多选后显示每个任务的选择按钮。
- 选择按钮位于任务最右侧。
- 支持全选。
- 支持反选。
- 支持批量设置高优先级。
- 支持批量设置正常优先级。
- 支持导出。

导出格式：

- JSON
- CSV
- Markdown

Markdown 导出会保留任务和子任务的复选框结构，适合迁移到其他 Markdown 工具。

### 3.10 待办回收站

待办删除采用软删除。

删除时保留：

- 标题
- 描述
- 状态
- 优先级
- 标签
- 截止时间
- 子任务
- 历史记录

恢复时：

- 清空 `deleted_at`。
- 保留删除前状态。
- 写入恢复历史。
- 重新出现在原有待办列表和相关筛选中。

### 3.11 待办在日历中的显示

待办可以显示在日历页面，但默认不显示。

规则：

- 日历页面有待办图层按钮。
- 默认状态下待办图层为取消状态。
- 用户点击待办图层后，日历渲染有截止时间的待办。
- 待办依据 `due_at` 显示在对应日期。
- 已完成和未完成待办在日期详情中可区分统计。
- 图层开关只影响展示，不修改任务数据。

## 4. 日历功能

### 4.1 日历定位

日历是 Flux 的时间聚合视图。

它不是单独替代日记、待办和事件的上级模块，而是读取这些模块的数据后按日期汇总展示：

- 日历事件
- 节假日
- 待办截止项
- 日记
- 已删除日记或事件标记

### 4.2 日历图层控制

日历使用图层控制不同内容是否展示。

当前默认状态：

| 图层 | 默认状态 | 说明 |
| :--- | :--- | :--- |
| 日历事件 | 选中 | 默认渲染事件。 |
| 节假日 | 选中 | 默认渲染节假日视觉标识。 |
| 待办 | 取消 | 用户点击后才渲染待办。 |
| 日记 | 取消 | 用户点击后才渲染日记。 |
| 回收站 | 取消 | 用户点击后才渲染删除标记。 |

规则：

- 选中时，日历看板实时渲染该类目数据。
- 取消时，日历看板隐藏该类目数据。
- 图层状态是前端展示状态，不会修改数据库。
- 日历事件和节假日默认更像日历基础层，所以默认选中。
- 日记和待办是生活记录和行动数据，为避免月视图过载，默认隐藏。

### 4.3 月视图

月视图用于快速查看一个月内每天是否有内容。

显示内容：

- 日期
- 是否为节假日
- 日历事件数量
- 待办数量
- 日记数量
- 删除标记数量

数据来源：

- 月聚合接口读取指定年月范围。
- 后端按日期聚合日记、待办、事件、删除项和节假日。
- 前端根据图层状态决定哪些内容实际渲染。

适用场景：

- 快速浏览某月记录密度。
- 找到有事件或有日记的日期。
- 识别假期和工作日。

### 4.4 日视图时间轴

日视图用于查看某一天的详细安排。

展示内容：

- 全天或有时间的日历事件。
- 有截止时间的待办。
- 当天日记。
- 节假日状态。

特点：

- 更接近时间轴。
- 适合查看当天事件和待办的先后关系。
- 日记作为当天记录展示，不强制进入时间轴排序。

### 4.5 周视图

周视图用于横向比较一周内每天的内容。

规则：

- 每天占一个日期格。
- 日记、待办、事件按紧凑卡片展示。
- 日记内容在周视图中需要控制长度，避免超出日期格子。
- 节假日作为日期背景或标识显示。

适用场景：

- 查看本周任务压力。
- 对比哪几天有记录或事件。
- 安排接下来几天的任务。

### 4.6 季度视图

季度视图用于更高层级的时间回顾。

规则：

- 一次显示三个月。
- 使用更摘要的方式展示事件、假期、日记和待办密度。
- 适合回顾季度记录节奏，而不是编辑具体内容。

### 4.7 那年今日

“那年今日”用于查看历史上同一月日发生过什么。

匹配规则：

- 选中某天后，按 `MM-DD` 匹配历史数据。
- 可匹配日记日期。
- 可匹配待办截止日期。
- 可匹配事件开始日期。

返回内容：

- 历史同日的日记。
- 历史同日的待办。
- 历史同日的事件。
- 汇总计数。

适用场景：

- 回看往年今天写过什么。
- 发现重复出现的纪念日或任务。
- 做个人时间线回顾。

### 4.8 那年这月

“那年这月”用于查看历史上同月份的数据。

匹配规则：

- 选中某天后，按月份 `MM` 匹配历史数据。
- 不限定具体日期。
- 覆盖日记、待办和事件。

适用场景：

- 回顾每年某个月的生活节奏。
- 查看某个季节性主题。
- 对比不同年份同月份的记录密度。

### 4.9 日期详情面板

用户点击日历中的日期后，会进入日期详情。

日期详情展示：

- 日期标题
- 节假日状态
- 事件数量
- 待办数量
- 日记数量
- 写日记、加待办、加事件入口
- 当天事件卡片
- 当天待办卡片
- 当天日记卡片
- 删除项标记或回收站内容

当一天同时存在日记、待办、事件时，三类内容需要通过标题、颜色、图标、卡片样式或标签进行区分，避免用户误认类型。

### 4.10 日历事件

日历事件是日历域自己的写入型数据。

事件包含：

- 标题
- 描述
- 开始时间
- 结束时间
- 是否全天
- 颜色
- 位置
- 提醒字段
- 重复规则字段
- 删除状态

当前说明：

- 事件可以在日历页面创建、编辑、删除和恢复。
- 删除后进入事件回收站逻辑。
- 事件默认在日历中展示，因为日历事件图层默认选中。
- 提醒字段可以保存，但当前不实现真实系统提醒通知。
- 重复规则字段存在，但当前不强调真正的重复任务生成能力。

### 4.11 回收站图层

日历可显示删除标记，但默认隐藏。

规则：

- 开启回收站图层后，日历可以显示当天是否存在删除日记或删除事件。
- 已经恢复或合并恢复的日记，不应继续造成不合理的删除标记。
- 恢复后日历聚合应重新计算，删除标记随数据状态消失。

## 5. 自定义节假日功能

### 5.1 节假日来源

Flux 的节假日判断由三层来源组成：

```text
用户手动覆盖
  > 静态节假日数据库
  > 默认周六周日
```

含义：

- 用户手动标记或取消拥有最高优先级。
- 静态节假日数据库用于保存法定节假日、调休等预置数据。
- 如果没有静态数据也没有用户覆盖，则周六周日默认是假期。

### 5.2 默认双休日假期

系统默认所有周六、周日是假期。

规则：

- 默认双休日不需要每一天写入用户覆盖表。
- 判断时根据日期计算星期。
- 如果用户没有手动覆盖，周六周日会显示为假期。

### 5.3 静态节假日数据库

项目已将静态节假日数据写入 SQLite。

用途：

- 标记法定节假日。
- 标记调休日或非假日工作日。
- 作为默认双休日规则之上的更精确日历数据。

特点：

- 运行时不再依赖外部 `holiday` 文件夹读取。
- 节假日查询直接走 SQLite 表。
- 更适合统一备份、迁移和部署。

### 5.4 手动标记节假日模式

用户可以在日历页面进入手动标记节假日模式。

交互规则：

- 开启模式后，用户点击任意日期。
- 如果该日期当前不是假期，则标记为假期。
- 如果该日期当前是用户覆盖出的假期，则再次点击可取消。
- 如果用户设置结果与系统默认值一致，后端会删除多余覆盖记录。
- 不需要为不同节假日设置不同名称。

视觉规则：

- 假期使用不同于普通日期的淡色背景或标识色。
- 手动标记模式下日期应有可点击反馈。
- 节假日按钮选中时应有不同颜色，方便识别当前模式。

### 5.5 节假日显示开关

节假日显示由日历页面的节假日功能控制。

规则：

- 节假日图层默认选中。
- 取消时隐藏节假日背景或标识。
- 选中时重新渲染节假日视觉。
- 显示开关不改变节假日数据。
- 手动标记节假日是数据操作，显示开关是渲染控制，两者概念不同但都在日历页完成。

## 6. 附件功能

### 6.1 支持类型

日记附件支持三类：

| 类型 | 用途 |
| :--- | :--- |
| 图片 | 日记插图、截图、照片。 |
| 音频 | 录音、语音备忘、声音片段。 |
| 普通文件 | PDF、文档、表格、Markdown、JSON、压缩包等。 |

普通文件支持常见办公文档和压缩格式。

### 6.2 上传规则

附件上传规则：

- 必须通过 `multipart/form-data` 上传。
- 文件不能为空。
- 文件大小必须小于 100MB。
- 扩展名必须在允许列表中。
- 文件保存在本地附件目录。
- 文件名使用系统生成 ID，避免重名覆盖。

保存路径：

```text
data/attachments/diaries/YYYY/MM/
```

返回给前端的是本地附件 URL：

```text
/attachments/diaries/YYYY/MM/<id>.<ext>
```

### 6.3 附件管理面板

设置中包含附件管理面板。

展示内容：

- 附件总数量
- 已引用数量
- 未引用数量
- 总大小
- 已引用大小
- 未引用大小
- 附件列表
- 附件类型
- 附件引用来源

支持筛选：

- 全部
- 图片
- 音频
- 普通文件

### 6.4 附件引用来源

附件管理会展示每个附件在哪里被引用。

引用来源包括：

- 日记 ID
- 日记日期
- 日记时间
- Markdown 所在行号
- 引用附近文本摘要

实现逻辑：

- 系统扫描所有未删除日记正文。
- 解析 Markdown 图片链接和普通链接。
- 如果链接指向 `/attachments/...`，则建立引用关系。

这种方式属于弱一致引用：

- Markdown 中存在链接，就认为被引用。
- 删除 Markdown 链接后，下一次扫描会认为附件未引用。
- 不额外维护强关系表。

### 6.5 单个附件删除

附件管理支持单个附件删除。

规则：

- 删除前根据 URL 定位本地文件。
- 后端校验路径必须位于附件根目录下。
- 删除时返回该附件的引用数量和引用来源。
- 删除文件不会自动修改日记 Markdown。
- 如果删除了仍被引用的附件，日记中会保留失效链接，后续需要用户自行处理。

### 6.6 未引用附件清理

附件管理支持清理未引用附件。

规则：

- 系统扫描附件目录。
- 系统扫描活跃日记 Markdown。
- 没有被任何活跃日记引用的文件会被识别为未引用。
- 用户触发清理后，删除未引用文件。
- 清理后尝试删除空目录。

适用场景：

- 删除日记内容后清理残留文件。
- 测试附件上传后清理无用资源。
- 保持本地数据目录轻量。

## 7. 导出功能

### 7.1 导出入口

导出能力分别存在于日记和待办。

通用交互：

- 点击导出或多选按钮。
- 进入多选状态。
- 默认全选当前列表中的条目。
- 用户可以单独选择或取消。
- 用户可以全选或反选。
- 用户选择导出格式。
- 系统生成下载内容。

### 7.2 日记导出

日记导出可按当前筛选或多选结果生成。

包含内容：

- 日期
- 时间
- 心情
- 天气
- 位置
- 收藏状态
- 字数
- 标签
- 正文
- 创建时间
- 更新时间

格式说明：

- JSON 保留结构完整性。
- CSV 方便表格工具查看。
- Markdown 更适合阅读和迁移。

### 7.3 待办导出

待办导出包含：

- 标题
- 描述
- 状态
- 优先级
- 标签
- 截止时间
- 子任务

Markdown 导出会使用复选框语法：

```markdown
- [ ] 未完成任务
- [x] 已完成任务
  - [ ] 子任务
```

### 7.4 导出限制

当前导出限制：

- 不自动打包附件二进制文件。
- Markdown 中保留附件路径。
- 导出是即时生成，不在数据库中保存导出任务。
- 没有导入恢复流程。

## 8. 回收站与恢复体系

### 8.1 统一软删除

以下对象采用软删除：

- 日记
- 待办
- 日历事件

软删除特点：

- 写入 `deleted_at`。
- 普通列表默认排除。
- 回收站或删除图层可查看。
- 可以恢复。
- 保留删除前状态。

### 8.2 各模块恢复差异

| 模块 | 恢复方式 | 特殊规则 |
| :--- | :--- | :--- |
| 日记 | 直接恢复或合并恢复 | 如果同日已有未删除日记，则增量合并。 |
| 待办 | 清空删除标记 | 保留删除前状态、优先级、标签、子任务。 |
| 日历事件 | 清空删除标记 | 恢复后重新进入事件图层。 |

### 8.3 恢复后的日历影响

恢复动作会影响日历聚合结果。

规则：

- 日记恢复后，当天日记计数恢复。
- 日记合并恢复后，来源记录不再制造删除标记。
- 事件恢复后，当天事件计数恢复。
- 删除标记应随恢复状态重新计算。
- 前端需要重新加载日历摘要或当天聚合数据。

## 9. 设置页功能

### 9.1 设置组织方式

设置项采用直接罗列方式。

原则：

- 可以分类，但不需要再通过额外标签页切换后才能看到设置。
- 常用操作应直接暴露。
- 附件管理作为设置中的重要面板展示。
- 节假日相关操作已移动到日历页面，不再放在设置页。

### 9.2 附件管理

设置中的附件管理负责数据治理。

能力：

- 查看附件数量和大小。
- 区分图片、音频、普通文件。
- 查看引用来源。
- 删除单个附件。
- 清理未引用附件。

## 10. 导航与页面结构

左侧导航栏顺序为：

```text
日记
日历
待办
```

说明：

- 日记代表记录系统，是个人内容中心。
- 日历代表时间聚合，是跨模块查看入口。
- 待办代表行动系统，是任务执行入口。

设置入口作为辅助治理入口，不作为核心业务顺序的一部分。

## 11. 时间与显示规则

### 11.1 本地时间展示

系统需要避免 UTC 时间直接暴露给用户造成误解。

规则：

- 数据库存储可以保留 ISO 时间。
- 用户界面应以本地时间理解和展示。
- 日记日期是语义日期，不应因为时区转换改变所属日期。
- 删除、恢复等操作显示时应符合用户当前时区认知。

### 11.2 日期字段和时间字段

日记：

- `entry_date` 是主日期。
- `entry_time` 是可选时间。

待办：

- `due_at` 决定日历中的待办展示日期。
- `completed_at` 记录完成时间。

事件：

- `start_at` 和 `end_at` 决定事件范围。
- `all_day` 决定是否全天展示。

节假日：

- 使用自然日判断。
- 用户手动覆盖按日期保存。

## 12. 当前未实现或弱化的能力

为了避免误解，当前以下能力没有作为完整功能实现：

| 能力 | 当前状态 |
| :--- | :--- |
| 用户账号与登录 | 未实现，本地单用户模型。 |
| 多用户权限 | 未实现。 |
| 真实系统提醒通知 | 未实现，仅保存提醒字段。 |
| 重复任务生成 | 已从产品范围中移除或弱化，不作为当前核心功能。 |
| “我的一天” | 已删除相关功能。 |
| 统计页面 | 已删除，不作为当前功能。 |
| 附件二进制打包导出 | 未实现，Markdown 只保留附件路径。 |
| 云同步 | 当前仅有同步预留思路，不是完整功能。 |
| 外部搜索引擎 | 未使用，当前日记搜索基于 SQLite FTS5。 |

## 13. 验收关注点

### 13.1 日记验收

- 同一天不能创建多篇未删除主日记。
- 编辑日期时不能改到已有日记日期。
- 删除日记后普通列表和搜索不再显示。
- 恢复日记时，同日无日记则直接恢复。
- 恢复日记时，同日已有日记则合并正文和标签。
- 合并恢复后，来源日记不再反复出现在回收站。
- 日记附件可以上传并插入 Markdown。
- 附件管理可以显示引用来源。

### 13.2 待办验收

- 任务支持正常和高优先级。
- 完成按钮可在完成和未完成之间切换。
- 已完成任务置底且变灰。
- 子任务完成后保持原位置。
- 多选模式下可以全选、反选和批量设置优先级。
- 删除任务后可以从回收站恢复。
- 恢复后保留删除前状态。

### 13.3 日历验收

- 默认显示事件和节假日。
- 默认隐藏待办和日记。
- 点击待办图层后显示有截止时间的待办。
- 点击日记图层后显示当天日记。
- 图层取消后对应内容隐藏但数据不变。
- 月、日、周、季度视图都可正常展示。
- 那年今日按 `MM-DD` 匹配历史数据。
- 那年这月按 `MM` 匹配历史数据。
- 恢复删除内容后日历删除标记应更新。

### 13.4 节假日验收

- 周六周日默认是假期。
- 静态节假日数据库中的日期按数据展示。
- 用户手动标记优先级高于默认和静态数据。
- 再次点击已手动标记日期可以取消或回到默认状态。
- 节假日图层隐藏时，假日视觉不渲染。

### 13.5 附件验收

- 图片、音频和普通文件能区分展示。
- 大于等于 100MB 的文件被拒绝。
- 不支持的扩展名被拒绝。
- 附件路径不能逃逸附件根目录。
- 单个附件可以删除。
- 未引用附件可以清理。
- 已引用附件应显示引用日记和行号。
