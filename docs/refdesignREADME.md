# Flux 设计理念与体验文档

## 1. 产品愿景

Flux 是一个围绕个人时间建立的本地生活管理空间。

它不是单纯的待办工具，也不是只用于写作的日记应用，更不是一个只展示事件的日历。Flux 试图把一个人的记忆、计划、行动和素材放在同一个安静、清晰、可掌控的界面里。

Flux 的核心判断是：

```text
一天不只是日期，它是生活的容器。
```

在 Flux 中，一天可以包含日记、日历事件、待办、节假日、附件和被恢复的历史。用户打开某一天时，看到的不只是安排，而是这一日的完整上下文。

## 2. 设计初衷

很多效率工具会让用户感觉自己被任务追赶。很多日记工具又把记录和行动完全分开。日历工具通常能告诉用户“什么时候有事”，却很少告诉用户“那一天发生了什么、留下了什么、我当时怎么想”。

Flux 想解决的是这种割裂。

它帮助用户回答几个朴素的问题：

- 今天有什么？
- 我写下了什么？
- 我还有什么没做？
- 哪些资料属于这一天？
- 哪些内容被删除过，又是否可以找回？

Flux 不追求制造更多操作，而是帮助用户把生活中的信息重新放回上下文里。

## 3. 核心体验原则

### 3.1 本地优先

日记、附件和任务都具有私人属性。Flux 默认将数据保存在本地，让用户知道自己的内容在哪里。

这种设计带来的感受是安全和安定：它更像放在自己桌面上的一本笔记本，而不是必须先信任远端服务才能使用的系统。

### 3.2 日期是主线

Flux 的信息组织以日期为核心。

一个日期可以连接：

- 一篇主日记
- 当天事件
- 当天待办
- 节假日状态
- 日记中的附件
- 回收站中的历史记录

日期不再是日历上的一个格子，而是用户回到某一天的入口。

### 3.3 先写下来，再整理

日记不要求标题。

真实写日记时，用户往往不是先构思标题，而是先写下一句话。Flux 让用户直接进入正文，减少开始记录时的阻力。

时间、心情、天气、位置、标签和附件都可以补充，但不是负担。用户可以认真记录，也可以只写几行。

### 3.4 删除应当可回头

Flux 的删除设计不是惩罚性的。

待办、日记和日历事件都支持回收站。用户可以恢复，也可以在同一天已有日记时，将删除日记合并回当天原日记。

这让“恢复”符合人的直觉：同一天的记忆应该可以重新汇聚，而不是制造冲突。

## 4. 视觉语言

Flux 的视觉气质是：

```text
安静、可靠、清晰、克制、有温度。
```

界面不追求强烈装饰，也不把数据做成压力很强的仪表盘。它更像一张整理过的工作台：工具在手边，内容在中央，用户可以自然地继续。

### 4.1 色彩语义

色彩用于表达内容类型，而不是制造视觉噪音。

| 内容类型 | 视觉方向 | 体验含义 |
| :--- | :--- | :--- |
| 日记 | 暖黄色 | 记忆、书写、个人痕迹 |
| 日历事件 | 蓝色 | 时间安排、确定性 |
| 待办 | 柔和红色 | 行动、任务、提醒 |
| 节假日 | 淡橙色 | 休息、例外、时间节奏 |
| 回收站 | 灰色 | 可恢复历史、降低强调 |
| 附件 | 类型标签 | 资料、证据、上下文 |

这种色彩策略让用户快速辨认信息来源，同时保持整体页面的低压感。

### 4.2 日期视觉强化

日记页面现在强化了日期视觉。

每篇日记卡片都有独立日期徽章，展示：

- 年份
- 月日
- 星期

这让日期从普通元信息变成了日记列表的时间锚点。用户滚动日记时，读到的不只是条目，而是一串清晰的日子。

### 4.3 标签视觉强化

标签不再和天气、位置、字数混在一起，而是以 `# 标签` 的胶囊形式独立展示。

这让标签更像主题线索。用户可以从一篇日记的标签跳转到对应筛选，也可以在侧栏看到每个标签关联了多少篇日记。

标签因此从“补充字段”变成“回看路径”。

### 4.4 布局逻辑

Flux 采用稳定的空间结构：

```text
左侧导航：模块切换
侧边区域：筛选和上下文控制
主内容区：当前工作对象
```

稳定结构的价值是减少重新学习。用户切换日记、日历、待办时，不需要重新理解页面。

## 5. 交互逻辑

### 5.1 导航顺序

当前左侧导航顺序是：

```text
日记
日历
待办
```

这体现了 Flux 当前的产品重心：先保存记忆，再理解时间，最后安排行动。

### 5.2 日历图层

日历默认展示：

- 日历事件
- 节假日

待办、日记和回收站内容需要用户主动打开。

这样设计是为了避免日历一开始就被所有信息填满。用户需要什么上下文，就打开什么图层。

### 5.3 日记写作路径

日记编辑器强调写作感。

它支持：

- Markdown
- 图片
- 音频
- 普通附件
- 时间
- 心情
- 天气
- 位置
- 标签
- 收藏

但这些能力不会阻挡用户写正文。正文区域是中心，元信息是辅助。

### 5.4 附件路径

附件不是孤立文件，而是日记的一部分。

用户可以把照片、录音、文档、表格或压缩包放进日记。附件管理面板再告诉用户：

- 文件是否还被引用
- 引用在哪篇日记
- 出现在第几行
- 是否可以清理

这让附件管理从“整理文件”变成“整理记忆材料”。

### 5.5 回收站路径

回收站承担的是安全感。

当用户删除一条日记、待办或事件时，内容不会立刻消失。恢复时系统尽量给出符合场景的结果。

尤其是日记：如果同一天已有一篇主日记，恢复另一篇删除日记时，Flux 会合并正文，而不是让用户面对覆盖、冲突或重复记录。

## 6. 核心使用场景

### 6.1 早晨：看今天

用户打开日历，先看到今天的事件和节假日状态。

如果需要行动上下文，可以打开待办图层。日历像一天的入口，帮助用户知道今天从哪里开始。

### 6.2 白天：收集材料

白天收到文件、拍下照片、录下音频时，用户可以把这些内容插入当天日记。

资料不再散落在文件夹里，而是和具体日期、具体文字放在一起。

### 6.3 晚上：写日记

用户进入日记页面，看到清晰的日期线索和标签主题。

写作时可以只写正文，也可以补充心情、天气、位置和附件。Flux 不要求记录必须完整，它允许用户按照当天的精力记录。

### 6.4 回看：按主题找自己

用户可以按收藏、心情、月份、年份和标签回看日记。

这种筛选不是为了管理数据，而是为了帮助用户理解自己的时间：哪些主题反复出现，哪些日子值得收藏，某段时间的情绪如何变化。

### 6.5 整理：清理附件

当附件越来越多时，用户进入附件管理。

系统会显示哪些附件仍被日记引用，哪些未被引用。删除前，用户能看到文件和日记之间的关系，因此整理过程更可控。

## 7. 品牌气质

Flux 的品牌感不来自夸张视觉，而来自稳定、私密和可回头。

它不催促用户打卡，也不把生活变成分数。它更像一个安静的个人工作室：

```text
你可以在这里安排事情，也可以保存记忆。
你可以计划未来，也可以重新理解过去。
```

理想状态下，用户打开 Flux 时感受到的是：

```text
我知道自己在哪里，也知道可以从哪里继续。
```

## 8. 当前体验风险

随着功能增多，需要持续注意：

- 日历图层过多时可能变得拥挤
- 日记编辑器的元信息可能挤压写作感
- 附件管理在文件增多后需要搜索、排序和批量能力
- 标签过多时可能从帮助变成负担
- 回收站、导出、附件清理等低频功能要避免干扰主路径

设计上应继续坚持：常用路径清晰，低频功能可达但不抢主舞台。

## 9. 未来体验方向

### 9.1 统一时间线

未来可以增加时间线视图，把日记、事件、待办和附件按时间汇聚。

用户不仅能看到“计划了什么”，也能看到“实际留下了什么”。

### 9.2 跨模块搜索

搜索可以从日记扩展到：

- 待办
- 事件
- 附件
- 标签

理想搜索不是只找到文本，而是把用户带回相关上下文。

### 9.3 数据安全感

备份、恢复和导出应该成为清晰的体验。

用户需要知道：

- 数据在哪里
- 如何备份
- 如何迁移
- 删除前会发生什么

### 9.4 更自然的写作

日记编辑可以继续向更自然的写作空间发展：

- 更好的 Markdown 预览
- 更舒服的长文阅读
- 更顺滑的附件插入
- 更清晰的媒体展示

## 10. 信息架构细节

Flux 当前的一级导航为：

```text
日记
日历
待办
设置
```

其中前三项构成用户的日常主路径：

| 页面 | 用户意图 | 主要操作 |
| :--- | :--- | :--- |
| 日记 | 记录、回看、搜索、按主题整理 | 写日记、筛选、搜索、导出、恢复 |
| 日历 | 理解时间、查看当天全貌 | 切换视图、选择日期、打开图层、标记节假日 |
| 待办 | 安排和推进任务 | 新建、完成、拖拽、筛选、恢复 |
| 设置 | 管理工具性能力 | 主题、附件管理、清理资源 |

设计重点是让高频路径短，低频路径不消失。

日记、日历、待办直接暴露在主导航里；附件管理放在设置中，是因为附件本身不是每天都要主动管理的对象，它更像维护能力。

## 11. 页面级体验说明

### 11.1 日记页

日记页由三部分组成：

```text
左侧筛选
顶部搜索与导出
日记卡片列表
```

左侧筛选强调“回看路径”：

- 收藏
- 心情
- 月份
- 年份
- 回收站
- 标签

卡片内的日期徽章让每篇日记拥有明确时间归属。标签胶囊让主题变得更可点击、更容易被再次使用。

这种设计让日记页既可以顺序阅读，也可以按主题回看。

### 11.2 日历页

日历页承担“总览”角色。

视图切换包括：

- 月
- 日
- 周
- 季度
- 那年今日
- 那年这月

内容图层包括：

- 日历事件
- 节假日
- 待办
- 日记
- 回收站

视图切换解决“用什么时间尺度看”，内容图层解决“看哪类信息”。这两个控制不应混在一起，因为它们回答的是不同问题。

### 11.3 待办页

待办页强调行动推进。

核心体验是：

- 快速看到未完成任务
- 完成后仍留在原上下文
- 完成任务置底而不是消失
- 子任务独立完成但不改变位置
- 高优先级只有一档，避免复杂优先级体系

待办页不追求复杂项目管理，而是服务个人日常行动。

### 11.4 设置页

设置页不再使用“先选中一个设置项再展开”的结构，而是直接罗列可操作内容。

这降低了操作层级，也符合当前设置项数量不大的阶段。

附件管理是设置页的核心功能之一，因为它承担长期数据卫生：

- 看见所有附件
- 区分图片、音频、其他
- 查看引用来源
- 删除单个附件
- 清理未使用附件

## 12. 组件语言

### 12.1 卡片

卡片只用于承载具体对象：

- 一条日记
- 一个事件
- 一个待办
- 一个附件

卡片不用于包装大区域，避免页面变成“卡片套卡片”的复杂视觉结构。

### 12.2 标签与 chip

Flux 中存在两类小型信息单元：

| 类型 | 用途 |
| :--- | :--- |
| 标签胶囊 | 表示主题，可点击，可筛选 |
| chip | 表示元信息，例如时间、心情、位置、字数 |

二者需要视觉区分。标签承担导航意义，chip 承担描述意义。

### 12.3 按钮

按钮遵循功能清晰原则：

- 高频主操作使用明确文字
- 删除使用危险色
- 恢复使用温和正向按钮
- 多选、导出、清理等低频操作放在工具区

按钮不应承担解释文案，界面也不应依赖大段说明才能理解。

### 12.4 日期徽章

日期徽章用于日记卡片，不用于所有模块。

原因是日记本质上是“时间中的文本”。强化日期能帮助用户把正文重新放回具体日子里。待办和事件已经有自己的时间表达方式，不需要同样的徽章。

## 13. 状态设计

### 13.1 空状态

空状态应该鼓励继续，而不是制造焦虑。

示例语气：

```text
还没有日记。
回收站是空的。
暂无附件。
没有定位到匹配的日记。
```

这些文案保持中性，不评价用户行为。

### 13.2 删除状态

删除状态使用灰色、虚线或弱化风格。

目标是表达“这条内容不在正常流程中，但仍可恢复”。它不能和正常内容一样强，也不能像错误一样刺眼。

### 13.3 完成状态

已完成待办应变灰，并留在原列表底部。

这个设计保留完成反馈，也保留任务曾经属于哪个列表的上下文。

### 13.4 选中状态

选中状态用于：

- 日历图层按钮
- 筛选标签
- 导出多选
- 附件类型筛选

选中态必须明确，因为它会改变页面展示内容。尤其是日历图层，用户需要知道为什么某些内容显示或隐藏。

## 14. 响应式体验

移动端设计重点不是塞下所有桌面内容，而是保持主路径可用。

当前策略：

- 主布局在窄屏下变为单列
- 左侧导航变为底部导航
- 日记卡片日期徽章在移动端横向压缩
- 周视图和紧凑卡片避免正文溢出
- 附件列表在移动端重新排列操作区

未来移动端可进一步优化：

- 日记筛选收进抽屉
- 日历图层按钮横向滚动
- 长文编辑器提供更大的全屏写作模式

## 15. 文案原则

Flux 的文案应该保持：

- 直接
- 温和
- 不评判
- 不制造紧迫感
- 不使用过度营销语言

推荐表达：

```text
日记已恢复
日记已合并到当天原日记
附件已插入
附件必须小于 100MB
已清理未使用资源
```

避免表达：

```text
你还没有完成任何任务
你的日记为空
立即清理
必须填写
```

产品语气应该像一个可靠的整理者，而不是一个监督者。

## 16. 可访问性与可读性

当前设计应继续关注：

- 文字对比度
- 按钮可点击面积
- 表单标签清晰
- 长文本不溢出容器
- 日期和标签在浅色/深色主题下都可辨认
- 颜色不作为唯一信息来源，必要时配合文字

对于日历来说，颜色区分之外仍要保留文字标记，例如：

```text
事
待
记
休
删
```

这样即使用户对颜色不敏感，也能理解格子中的内容类型。

## 17. 设计验收清单

| 检查项 | 标准 |
| :--- | :--- |
| 日记卡片 | 日期醒目，标签独立，正文可读，操作不拥挤 |
| 日历 | 默认不拥挤，图层状态明确，节假日可识别 |
| 待办 | 完成、恢复、删除、优先级含义明确 |
| 附件 | 类型区分清楚，引用来源可理解，删除前有依据 |
| 回收站 | 与正常内容有区分，但恢复入口明显 |
| 移动端 | 主路径可操作，文本不重叠，不出现横向挤压 |
| 深色主题 | 日期、标签、按钮、chip 仍可读 |

## 18. 总结

Flux 是一个关于个人连续性的产品。

它连接计划和记忆，行动和反思，文件和情绪。它的设计目标不是让用户做更多，而是让用户更清楚地看见自己的生活。

Flux 最好的状态不是一个指挥中心，而是一个可以反复回来的地方。
